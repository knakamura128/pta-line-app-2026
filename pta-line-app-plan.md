# PTA活動募集サイト 設計メモ

## 1. 目的

LINE公式アカウント上で利用できる、PTA向けの活動募集サイトを構築する。
利用者はLINEミニアプリ(LIFF)から募集一覧を見て回答し、管理者は募集作成、複製、締切管理、通知を行う。

## 2. 想定ユーザー

### 回答者
- 保護者
- LINE公式アカウントを友だち追加済み
- LIFFから募集ページを開き、ログイン済み状態で回答する

### 管理者
- PTA本部役員、学校側担当者
- 管理画面から募集を作成、公開、締切管理、決定通知を行う

## 3. 要件整理

### 回答者向け要件
- LIFFで起動できる
- LINEログイン情報で利用者を識別できる
- 公開中の募集アンケートを一覧表示できる
- 募集ごとに詳細を確認し回答できる
- 回答内容は締切まで編集できる
- 回答人数を表示できる
- 他の回答者の氏名は表示しない
- 自分の回答状況を確認できる
- 締切後は回答不可にする
- 募集人数に達した時点で新規回答を停止する

### 管理者向け要件
- 募集アンケートを簡単に作成できる
- 既存募集からコピー新規できる
- 公開期間を設定できる
- 募集人数を設定できる
- スタッフ区分、運営区分を選択肢として設定できる
- 開催日、開催時間帯を設定できる
- 募集内容、お仕事内容を記載できる
- 回答受付完了通知を送信できる
- 締切後に確定通知を送信できる
- 確定通知の本文を管理画面で設定できる
- 確定通知の本文はアンケート作成時点で入力必須にできる
- 決定者へ個別にLINE通知できる
- 回答状況と定員充足状況を確認できる

## データベース

## インフラ候補
- 小規模スタートなら `Next.js + Supabase + Vercel`
- もう少し堅くやるなら `Next.js + PostgreSQL + Node.js API + cronワーカー`

## 7. 推奨技術スタック

実装速度と運用コストを優先する場合:

- フロント/管理画面: Next.js
- UI: Tailwind CSS
- 認証連携: LINE Login + LIFF SDK
- DB: PostgreSQL
- ORM: Prisma
- バックグラウンド: Vercel Cron または GitHub Actions、必要なら separate worker
- 通知: LINE Messaging API

理由:
- LIFFとWeb管理画面を同じアプリ基盤で持ちやすい
- 募集管理のようなCRUD中心機能に相性がよい
- 初期構築と保守の負荷を抑えやすい

## 8. 認証、権限設計

### 回答者
- LIFF起動後に `LINE userId` を取得してユーザー識別
- 初回アクセス時にユーザーレコードを作成
- 表示名は取得できるが、回答者一覧には使わない

### 管理者
- 管理者は別権限テーブルで管理
- `LINE userId` が管理者一覧に登録されている場合のみ管理画面にアクセス可能
- 運用上はメール認証よりLINEベースの方がシンプル
- LINE OAの管理者権限をそのままアプリへ自動同期する仕組みは標準では扱いにくいため、MVPではアプリ側の管理者一覧を別管理にするのが現実的
- まずは単独管理者1名で開始し、後から複数管理者対応へ拡張する方針が安全

7. 通知実行

## 11. 主要業務フロー

### 募集作成
1. 管理者が募集を新規作成
2. 公開期間、開催日時、募集人数、内容を入力
3. 公開状態に変更
4. LIFF側一覧に表示

### 回答
1. 回答者がLINE公式アカウントからLIFFを起動
2. 募集一覧から案件を選択
3. 詳細確認後に回答
4. 定員未到達なら受付し、回答人数が更新される
5. 回答受付完了メッセージをLINEで送る
6. 締切前であれば回答内容を編集できる
7. 締切前であれば回答を取り消しできる
8. 取り消し後は現在回答人数が減り、定員未達なら回答ボタンを再度活性化する

### 先着確定
1. 回答受付時に定員到達チェックを行う
2. 定員枠内に入った回答者を参加確定扱いにする
3. 定員に達した時点で募集状態を自動終了にする
4. 締切後すぐ、または数分後に確定者へLINE Messaging APIで個別通知する
5. 確定通知には管理画面で設定した本文を使用する
5. 回答取り消しが出ても再募集は行わない

## 12. 通知設計の注意点

- LINE公式アカウントからの個別通知は、友だち追加済みユーザーが前提
- 配信対象条件は明確に分ける
  - 回答受付完了通知
  - 締切後の確定通知
- 同一通知の二重送信防止が必要
- テンプレート文面は管理画面から編集できると運用が楽
- 確定通知本文はアンケート作成時に `text` として保存し、必要なオープンチャットURLやパスワードをそのまま含める
- 先着終了方式では、定員到達直後の通知遅延や二重確定を防ぐため排他制御が必要
- 回答取り消しを許可する場合、回答レコード削除と `is_confirmed` の整合を保つ必要がある

## 13. 回答者の個人情報表示方針

- 回答者画面には総回答人数のみ表示
- 他者の氏名、学年、連絡先は表示しない
- 管理者画面のみ個人識別情報を表示
- 個人情報保護方針と利用目的の明記が必要

## 14. 非機能要件

- スマホ中心で使いやすいUI
- LINE内ブラウザで安定動作
- 期限判定を日本時間で統一
- 誤送信防止の確認画面
- 通知履歴、操作履歴の保持
- 将来的な学校別、年度別のデータ分離を考慮

## 15. 実装時の論点

### 回答単位
- 回答単位は個人
- `LINE userId` をそのまま回答者識別子に使う
- 家庭単位制約は現時点では不要

### スタッフ、運営選択の粒度
- 単一選択か複数選択か
- 固定マスタか募集ごと自由入力か

### 決定方式
- 先着順自動決定
- 募集人数に達した時点で自動終了
- 回答編集は締切まで可能
- 回答済みユーザーの取り消しは締切まで許可
- 取り消しが出た場合は現在人数だけ減らし、定員未達なら回答ボタンを再活性化する
- 確定通知は締切後に送る
- 回答直後には受付完了通知を送る
- 確定通知の送信は自動で、締切直後または10分後程度に送る
- 確定通知の文面はアンケート作成時に管理画面で入力した `text` を使う

## 17. まず確定したい仕様

以下は現時点の確定仕様。

1. 回答単位は個人
2. 決定方式は先着順で、募集人数到達時に自動終了
3. 回答は締切まで編集可能
4. 回答済みユーザーは締切まで回答取り消し可能
5. 管理者権限は理想はLINE OA管理者連動だが、MVPではアプリ独自管理を前提にする
6. 回答取り消しが出たら現在人数を減らし、定員未達なら回答ボタンを再活性化する
7. 確定通知は締切後に送る
8. 回答直後は受付完了通知を送る
9. 確定通知は自動送信し、文面はアンケート作成時に管理画面で入力した `text` を使う
10. 確定通知の自動送信は締切直後または10分後程度に実行する

## 18. 追加で詰めるべき仕様

先着順運用では、以下を次に決める必要がある。

1. 回答編集で変更できる項目をどこまで許可するか
2. 受付完了通知と確定通知の文面テンプレートをどう分けるか
3. 締切後の自動送信を即時実行にするか、10分後固定にするか
